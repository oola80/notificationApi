1. Descripción General

La Notification API es una plataforma centralizada y orientada a eventos que consolida todas las notificaciones al cliente (Email, WhatsApp, SMS y Push) en un único sistema de microservicios. Los sistemas fuente se integran mediante webhook REST o AMQP directo; el motor de mapeo en tiempo de ejecución normaliza sus payloads al formato canónico sin cambios de código.

✓
Propuesta de Valor Los sistemas fuente se convierten en simples publicadores de eventos. Toda la inteligencia de notificación — qué enviar, cuándo, a quién y por qué canal — vive en la Notification API y se gestiona a través de una interfaz de administración centralizada. Nuevas fuentes pueden incorporarse únicamente mediante configuración, sin cambios de código.

Principios Arquitectónicos

PrincipioDescripción
DesacoplamientoLos servicios se comunican a través de APIs bien definidas y mensajes asíncronos. Ningún servicio accede directamente a la base de datos de otro.
Responsabilidad ÚnicaCada microservicio posee un único contexto: ingesta de eventos, orquestación, renderizado de plantillas, enrutamiento de canal, administración o auditoría.
Event-DrivenLos eventos de negocio fluyen por RabbitMQ, desacoplando productores de consumidores con entrega garantizada y procesamiento asíncrono resiliente.
Base de Datos por ServicioCada servicio posee su propio esquema PostgreSQL. El aislamiento de datos garantiza independencia en la evolución del esquema.
Resiliencia por DefectoCircuit breakers, políticas de reintento, colas de mensajes muertos (DLQ) y health checks están incorporados en cada servicio.

4. Microservicios

Servicio Puerto Responsabilidad Comunicación
notification-gateway 3150 API Gateway / BFF — autenticación, autorización, rate limiting y routing al backend. REST
event-ingestion-service 3151 Recibe eventos de cualquier fuente, aplica mapeos en tiempo de ejecución y normaliza al formato canónico. RabbitMQ REST
notification-engine-service 3152 Orquestador central — evalúa reglas, resuelve destinatarios y coordina renderizado y entrega. RabbitMQ REST
template-service 3153 CRUD de plantillas Handlebars con versionado y variantes por canal (Email / SMS / WhatsApp / Push). REST
channel-router-service 3154 Enruta mensajes renderizados al proveedor correcto. Implementa retry exponencial y circuit breakers. RabbitMQ Proveedores
admin-service 3155 Backend del backoffice — gestiona reglas, plantillas, grupos de destinatarios y configuración del sistema. REST
audit-service 3156 Registra el ciclo de vida completo de cada notificación: intentos, recibos del proveedor y analítica. RabbitMQ REST
email-ingest-service 3157 / 2525 Ingest SMTP para sistemas legacy — recibe emails, los parsea y genera eventos estándar. SMTP RabbitMQ
bulk-upload-service 3158 Procesa archivos XLSX del Admin UI, genera eventos por fila y los envía al servicio de ingesta. REST
notification-admin-ui 3159 Frontend Next.js — dashboards, editor de plantillas, configuración de reglas y visor de auditoría. REST

5. Stack Tecnológico

Backend y Frontend

  • NestJS + TypeScript — todos los microservicios
  • Next.js — Admin UI (SSR + React)
  • PostgreSQL — base de datos relacional (schema-per-service)

Infraestructura

  • RabbitMQ — broker de mensajes AMQP (6 exchanges, 23 colas)
  • Docker / Docker Compose — contenedorización y orquestación local

Proveedores de Entrega

  • Email: SendGrid, Mailgun, Braze
  • SMS / WhatsApp: Twilio, Braze
  • Push: Firebase Cloud Messaging (FCM), Braze

Testing y Observabilidad

  • Jest — pruebas unitarias (≥80% cobertura por servicio)
  • Playwright — E2E del Admin UI
  • Prometheus + Pino — métricas y logging estructurado

6. Flujo de Eventos

#PasoServicioDescripción
1PublicaciónSistema FuenteEl sistema fuente publica un evento de negocio (ej. "order.shipped") vía AMQP o webhook REST.
2Ingestaevent-ingestion-serviceValida el payload, aplica el mapeo en tiempo de ejecución y normaliza al formato canónico. Asigna prioridad (normal / crítico).
3Publicación InternaRabbitMQEl evento normalizado se publica al exchange xch.events.normalized con routing key por prioridad y tipo.
4Orquestaciónnotification-engine-serviceEvalúa reglas de notificación, resuelve destinatarios y preferencias de canal. Solicita renderizado al Template Service.
5Renderizadotemplate-serviceResuelve la versión de plantilla correcta, sustituye variables con datos del evento y retorna contenido por canal.
6Enrutamientochannel-router-serviceRecibe el mensaje renderizado y lo envía al proveedor correspondiente. Aplica retry exponencial y circuit breakers.
7Auditoríaaudit-serviceRegistra cada paso del proceso: intentos, recibos del proveedor, analítica y logs de auditoría para cumplimiento.
ⓘ
Patrón de resiliencia Todos los pasos son asíncronos vía RabbitMQ, excepto el renderizado de plantillas (REST síncrono) y la entrega al proveedor (HTTP síncrono). Cada cola tiene una DLQ asociada. Los mensajes fallidos tras 3 intentos son enrutados a xch.notifications.dlq para revisión manual o reprocesamiento automático.
