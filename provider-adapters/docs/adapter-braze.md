# Braze Adapter Reference

**adapter-braze — Port 3172**

---

## Overview

The Braze adapter handles multi-channel delivery (Email, SMS, WhatsApp, Push) via Braze's `/messages/send` REST API. It operates as a **stateless dumb pipe** — all content (HTML email bodies, SMS text, WhatsApp template parameters) is **pre-rendered by the template-service** before reaching the adapter. Braze is used purely as a delivery transport; it does not perform any template rendering, personalization, or content assembly. Braze is the only adapter supporting all four notification channels from a single integration.

The adapter includes a profile sync module to ensure user profiles exist in Braze before sending (see [Profile Sync Module](#profile-sync-module)).

| Attribute | Value |
|---|---|
| **Provider** | Braze |
| **Channels** | Email, SMS, WhatsApp, Push |
| **Port** | 3172 |
| **SDK/Client** | HTTP REST (`axios` calls to Braze REST API) |
| **Auth** | Bearer token (`Authorization: Bearer ${BRAZE_API_KEY}`) |
| **Estimated LOC** | ~500 |

## Braze External ID — Deep Dive

### What Is `external_id`?

Braze's `external_id` is the **primary user identifier** — a string you assign when creating a user profile in Braze. It is permanent, immutable per profile, and is the main way to target users for messaging. Every Braze user profile has exactly one `external_id` (or none, if the profile is anonymous/alias-only).

> **Key point:** `external_id` is NOT generated by Braze. It is YOUR system's unique identifier for the user, set at profile creation time.

### Identifier Types in Braze

| Identifier | Set By | Mutability | Use Case |
|---|---|---|---|
| `external_id` | Your system (at profile creation) | Immutable | Primary targeting for `/messages/send` — maps to your user/customer ID |
| `braze_id` | Braze (auto-generated) | Immutable | Internal Braze identifier; useful for anonymous users before identification |
| `user_alias` | Your system (label + name pair) | Mutable | Alternative targeting when `external_id` is unavailable (e.g., pre-registration users) |

### Mapping to Our System — Hashed Email as `external_id`

In our notification platform, we use the **recipient's email address** as the basis for the Braze `external_id`. For privacy and security, the email is **not stored in plaintext** — instead, it is hashed before being used as the external ID:

```
recipient.email  -->  SHA-256(pepper + normalize(email))  -->  Braze external_id
```

#### Hashing Specification

| Attribute | Value |
|---|---|
| **Algorithm** | SHA-256 (Node.js `crypto.createHash('sha256')`) |
| **Input normalization** | Trim whitespace, Unicode NFKC normalization, lowercase |
| **Pepper** | Secret pepper prepended before hashing: `sha256(pepper + normalizedEmail)` |
| **Output** | Hex-encoded SHA-256 digest (64 characters) |
| **Pepper source** | `EMAIL_HASH_PEPPER` environment variable |
| **Pepper caching** | In-memory cache with configurable TTL (`PEPPER_CACHE_TTL`) |

#### Hashing Endpoint

The hashing is performed by a dedicated endpoint in the platform:

```
POST /v1/customers/hash-email
Authorization: Bearer <token>
```

**Input:** Email address (normalized in the DTO — trimmed, NFKC, lowercased)

**Response:**
```json
{
  "email_hash": "a1b2c3d4e5f6...64-char-hex-digest",
  "algo": "SHA-256",
  "algo_version": "v1",
  "normalized_email": "user@example.com"
}
```

> **Key point:** The `email_hash` value is what gets used as the Braze `external_id`. This ensures privacy-safe identity matching — the actual email cannot be reverse-engineered from the hash without the secret pepper, protecting against rainbow table attacks.

#### Why Hashed Email?

- **Privacy:** Braze stores the `external_id` in plaintext. Hashing prevents email exposure if Braze data is compromised.
- **Deterministic:** The same email always produces the same hash (given the same pepper), so lookups and deduplication work reliably.
- **Cross-system matching:** Other systems can compute the same hash (with the pepper) for identity matching without sharing raw emails.

### The User Existence Prerequisite

Unlike Mailgun (fire-and-forget to any email) or Meta WhatsApp (fire-and-forget to any phone number), **Braze requires the user to exist in its system before you can send them a message**. If you call `/messages/send` with an `external_id` that Braze doesn't recognize, the message silently fails (returns 201 but with `errors` array indicating no valid users).

This has fundamental implications for the adapter's architecture — see [Profile Sync Module](#profile-sync-module) below.

### DTO Gap and Planned Fix

The current adapter-facing `RecipientDto` (in `libs/common`) only has two fields:

```typescript
// Current RecipientDto — libs/common/src/dto/send-request.dto.ts
export class RecipientDto {
  address: string;  // email, phone, or device token
  name?: string;    // display name
}
```

The channel-router's `toAdapterPayload()` maps `address` (from the channel-specific field: `email`, `phone`, or `deviceToken`) and `name`, but does **NOT** forward `customerId`.

**Planned fix:** Add an optional `customerId` field to `RecipientDto`:

```typescript
export class RecipientDto {
  address: string;
  name?: string;
  customerId?: string;  // Braze external_id (hashed email); other adapters ignore
}
```

And update the channel-router's `toAdapterPayload()` to forward it:

```typescript
recipient: {
  address,
  name: request.recipient.name,
  customerId: request.recipient.customerId,  // NEW — hashed email for Braze
},
```

Other adapters (Mailgun, WhatsApp/Meta, AWS SES) simply ignore the field.

## Braze Particularities & Strategy

Braze differs significantly from our other providers. This section documents the key differences and the strategy for handling them.

### 0. Braze as Dumb Pipe — We Render, Braze Delivers

Like all adapters in our platform, Braze operates as a **dumb pipe**. The adapter receives fully rendered content and passes it through to Braze for delivery:

| Responsibility | Owner | NOT Braze |
|---|---|---|
| Template storage & versioning | template-service | Braze templates are NOT used |
| Template rendering (Handlebars) | template-service | Braze Liquid/personalization is NOT used |
| Recipient resolution | notification-engine-service | Braze segments/audiences are NOT used |
| Channel routing & fallback | channel-router-service | Braze Canvas/campaigns are NOT used |
| Content delivery | **Braze** (via `/messages/send`) | — |
| User profile sync | **Braze adapter** (via `/users/track`) | — |

> **Key point:** Braze is configured to accept pre-rendered content only. No Braze-side templates, campaigns, Canvases, or Liquid personalization are used. All intelligence stays in our platform.

### 1. User Existence Prerequisite

| Provider | Model | Implication |
|---|---|---|
| Mailgun | Fire-and-forget to any email address | No user management needed |
| Meta WhatsApp | Fire-and-forget to any phone number | No user management needed |
| AWS SES | Fire-and-forget to any email address | No user management needed |
| **Braze** | **User must exist before send** | **Profile sync required** |

This is the most impactful difference. The adapter must either ensure users exist (just-in-time sync) or rely on an external process to pre-provision them. See [Profile Sync Module](#profile-sync-module) for strategies and trade-offs.

### 2. `app_id` Required for All Channels

Every message object in the `/messages/send` payload **requires** `app_id`. This identifies the Braze workspace/app. It is set globally via the `BRAZE_APP_ID` environment variable and injected into every channel message object.

```
messages.email.app_id       = BRAZE_APP_ID
messages.sms.app_id         = BRAZE_APP_ID
messages.whatsapp.app_id    = BRAZE_APP_ID
messages.apple_push.app_id  = BRAZE_APP_ID
messages.android_push.app_id = BRAZE_APP_ID
```

### 3. Subscription Group Management (SMS & WhatsApp)

Braze uses subscription groups to manage opt-in/opt-out per channel. SMS and WhatsApp **require** a `subscription_group_id` in the message object. The adapter reads these from environment variables:

- SMS: `BRAZE_SMS_SUBSCRIPTION_GROUP`
- WhatsApp: `BRAZE_WHATSAPP_SUBSCRIPTION_GROUP`

If the subscription group is not set, the send will fail with a 400 error.

### 4. WhatsApp: Braze vs Meta Native API

The Braze adapter and the WhatsApp (Meta) adapter both support WhatsApp, but their APIs differ significantly:

| Aspect | Meta Cloud API (adapter-whatsapp) | Braze `/messages/send` (adapter-braze) |
|---|---|---|
| **Targeting** | Phone number directly | `external_id` (user must exist) |
| **Template dispatch** | `type: "template"` + `template.name` + `template.language` + `template.components` | `message_type: "template_message"` + `message.template_name` + `message.template_language_code` + `message.variables` |
| **Text messages** | `type: "text"` + `text.body` | `message_type: "text_response_message"` + `message.body` |
| **Template parameters** | Positional components (header, body, button) with indexed parameters | Named variable objects: `{ "key": "variable_name", "value": "variable_value" }` |
| **Media in templates** | Header component with `type: "image"/"video"/"document"` + handle/link | IMAGE header only via `/messages/send` API (video/document not supported via API) |
| **Subscription** | Managed externally (Meta Business Manager) | `subscription_group_id` required in payload |

> **Warning:** **WhatsApp Routing Rule:** Both adapter-whatsapp and adapter-braze support WhatsApp. Only one can be active for the WhatsApp channel at a time — controlled via the `isActive` flag on the provider registration in channel-router-service.

### 5. WhatsApp `message_type` Enum

Braze WhatsApp uses a `message_type` field (not a `type` field like Meta):

| `message_type` Value | Description |
|---|---|
| `template_message` | HSM template message (outbound, requires pre-approved template) |
| `text_response_message` | Plain text reply (within 24-hour conversation window) |
| `image_response_message` | Image reply |
| `quick_reply_response_message` | Quick reply buttons |

### 6. Media Limitations via API

When sending WhatsApp template messages via the Braze `/messages/send` API:
- **IMAGE** headers are supported
- **VIDEO** and **DOCUMENT** headers are **NOT supported** via the REST API (only via Braze dashboard campaigns)

This is a known Braze API limitation. If video/document template headers are needed, use the Meta native adapter (adapter-whatsapp) instead.

### 7. Multi-Channel from Single Adapter

Braze is unique among our adapters in supporting all four channels from one API. The adapter must inspect the `channel` field in the `SendRequestDto` and build the appropriate channel-specific message object (`messages.email`, `messages.sms`, `messages.whatsapp`, `messages.apple_push` / `messages.android_push`).

### 8. Email `from` Field Format

Braze email requires a `from` field in the format `"Display Name <email@domain.com>"`. The adapter constructs this from:
- `recipient.name` or `BRAZE_FROM_NAME` for the display name
- `BRAZE_FROM_EMAIL` for the email address

## API Endpoints

| Endpoint | Description |
|---|---|
| `POST {BRAZE_REST_ENDPOINT}/messages/send` | Send notification (all channels) |
| `POST {BRAZE_REST_ENDPOINT}/users/track` | Sync user attributes/profile |
| `GET {BRAZE_REST_ENDPOINT}/subscription/status/get` | Check subscription group status |

### Braze REST Endpoint

The base URL is region-specific (e.g., `https://rest.iad-01.braze.com`, `https://rest.iad-03.braze.com`). Set via `BRAZE_REST_ENDPOINT` environment variable.

### `/messages/send` Key Facts

- Accepts `external_user_ids` array (up to 50 per request) — NOT `external_ids`
- User MUST already exist in Braze; unknown IDs silently fail (201 with errors)
- Supports `user_aliases` as an alternative to `external_user_ids`
- Returns `dispatch_id` on success
- Rate limit: 250,000 requests/hour per workspace

### `/users/track` Key Facts

- Creates or updates user profiles
- Requires at least one identifier: `external_id`, `user_alias`, `braze_id`, `email`, or `phone`
- Supports `_update_existing_only` flag (default `false`) — set to `true` to prevent accidental profile creation
- Rate limit: **3,000 requests per 3 seconds** per workspace
- Batches up to 75 attributes objects per request

## Profile Sync Module

Braze requires a user profile with a valid `external_id` to exist before messages can be sent. The adapter includes a profile sync module that ensures profiles exist via the `/users/track` API.

### What `/users/track` Does

The Braze `/users/track` endpoint **creates or updates** user profile attributes. The adapter uses it to:

1. **Create the profile** with the hashed email as `external_id` (if it doesn't exist)
2. **Set standard profile attributes** — `email` and `phone` are first-class Braze profile fields (not custom attributes). Setting them enables Braze to deliver via email and SMS/WhatsApp channels.

```json
{
  "attributes": [{
    "external_id": "a1b2c3...hashed-email",
    "email": "user@example.com",
    "phone": "+50212345678"
  }]
}
```

> **Key point:** `email` and `phone` are **standard Braze user profile attributes**. When set, Braze automatically associates them with the user profile for channel delivery. The adapter sets whichever is available from `recipient.address` based on the notification channel.

### Strategy Comparison

| Strategy | Description | Pros | Cons |
|---|---|---|---|
| **Just-in-time sync** (`/users/track` before send) | Adapter calls `/users/track` before every send (with LRU cache to avoid redundant calls) | Self-contained; no external dependencies; guaranteed user exists | Adds latency (~50-100ms per uncached call); `/users/track` rate limit (3,000/3s) can bottleneck high-volume sends; creates minimal profiles |
| **Pre-provisioned** (external sync) | A separate process or the source system provisions users in Braze ahead of time | Zero send-time overhead; richer profiles (all attributes synced); no rate limit pressure at send time | Requires external orchestration; risk of send failure if user not yet synced; eventual consistency gap |

### Just-in-Time Sync Flow (Default)

1. Receive `POST /send` with `recipient.customerId` (hashed email) and `recipient.address` (email or phone)
2. Check LRU cache for `customerId`
3. If cache miss: call `POST /users/track` to create/update the profile:
   - `external_id` ← `recipient.customerId` (the hashed email)
   - `email` ← `recipient.address` (when channel is email)
   - `phone` ← `recipient.address` (when channel is SMS or WhatsApp)
4. Cache the `customerId` (TTL configurable via `BRAZE_PROFILE_CACHE_TTL_SECONDS`, default 3600)
5. Proceed with `/messages/send` using `external_user_ids: [customerId]`

> **Info:** The `/users/track` call is idempotent. If the profile already exists, it updates the attributes. If it doesn't exist, it creates a new profile. This means the just-in-time sync is safe to call repeatedly.

### Pre-Provisioned Flow

1. External process syncs users to Braze (e.g., nightly batch or real-time event-driven)
2. Adapter receives `POST /send` with `recipient.customerId` (hashed email)
3. Adapter calls `/messages/send` directly using `external_user_ids: [customerId]`
4. If send fails with "no valid users" error, return appropriate error to channel-router

> **Note:** Profile sync is toggled via `BRAZE_PROFILE_SYNC_ENABLED`. When `true` (default), the adapter uses just-in-time sync. When `false`, it assumes the user already exists and skips `/users/track`.

## Subscription Groups

Braze uses subscription groups to manage opt-in/opt-out compliance per channel. SMS and WhatsApp messages **require** a `subscription_group_id` in the payload — sends without it return a 400 error.

### Approach: Static Single-Group

The adapter uses a **static, single subscription group per channel**, configured via environment variables:

| Channel | Environment Variable | Example |
|---|---|---|
| SMS | `BRAZE_SMS_SUBSCRIPTION_GROUP` | `sub-group-sms-123` |
| WhatsApp | `BRAZE_WHATSAPP_SUBSCRIPTION_GROUP` | `sub-group-wa-123` |

The subscription group ID is injected into every SMS/WhatsApp message payload automatically. This approach assumes all recipients belong to a single subscription group per channel — appropriate for a centralized notification platform where opt-in/opt-out is managed uniformly.

> **Note:** Email and Push channels do NOT require subscription group IDs in the `/messages/send` payload. Braze manages email subscription status at the user profile level and push at the device token level.

## Webhook Events — Transactional Postbacks vs Currents

Braze provides two webhook mechanisms:

| Mechanism | Latency | Channels | Use Case |
|---|---|---|---|
| **Transactional Postbacks** | Real-time (~seconds) | Email only | Immediate delivery status for transactional emails |
| **Currents** | Batched (~5 minutes) | All channels | Comprehensive event streaming (all engagement events) |

The adapter handles both:
- **Transactional postbacks** arrive at `POST /webhooks/inbound` as individual JSON events
- **Currents** events arrive as batched JSON payloads (multiple events per request)

### Webhook Event Types

| Braze Event | Mapped Status | Source | Description |
|---|---|---|---|
| `users.messages.email.Delivery` | `DELIVERED` | Postback/Currents | Email delivered to inbox |
| `users.messages.email.Open` | `OPENED` | Currents | Email opened |
| `users.messages.email.Click` | `CLICKED` | Currents | Link clicked |
| `users.messages.email.Bounce` | `BOUNCED` | Postback/Currents | Hard bounce |
| `users.messages.email.SoftBounce` | `TEMP_FAIL` | Postback | Soft bounce (may retry) |
| `users.messages.email.SpamReport` | `SPAM_COMPLAINT` | Currents | Marked as spam |
| `users.messages.email.Unsubscribe` | `UNSUBSCRIBED` | Currents | Unsubscribed |
| `users.messages.sms.Delivery` | `DELIVERED` | Currents | SMS delivered |
| `users.messages.sms.Rejection` | `BOUNCED` | Currents | SMS rejected by carrier |
| `users.messages.whatsapp.Read` | `READ` | Currents | WhatsApp message read |
| `users.messages.whatsapp.Delivery` | `DELIVERED` | Currents | WhatsApp delivered |
| `users.messages.whatsapp.Failure` | `BOUNCED` | Currents | WhatsApp delivery failed |

## Webhook Verification

Braze webhook verification depends on the mechanism:
- **Transactional postbacks:** Verified via shared secret header (`X-Braze-Webhook-Key` matching `BRAZE_WEBHOOK_KEY`)
- **Currents:** Currents connector verification is configured in the Braze dashboard (IP allowlisting + shared secret)

## Rate Limits

| Endpoint | Limit | Notes |
|---|---|---|
| `/messages/send` | 250,000 requests/hour | Per workspace, shared with other API calls |
| `/users/track` | 3,000 requests/3 seconds | Per workspace; applies to profile sync module |

> **Note:** The `/users/track` rate limit (3,000/3s) is the binding constraint for just-in-time profile sync under high volume. The LRU cache mitigates this for repeat recipients.

## Error Classification

| HTTP Status | Error Type | Retryable | Notes |
|---|---|---|---|
| 201 | Success | — | Message sent successfully |
| 400 | Bad Request | No | Invalid parameters or missing `external_id` |
| 401 | Unauthorized | No | Invalid API key |
| 429 | Rate Limited | Yes | Respect `Retry-After` header |
| 500 | Server Error | Yes | Braze internal error |
| 503 | Unavailable | Yes | Service temporarily unavailable |

> **Warning:** Braze returns **201** even when some recipients fail. Always check the `errors` array in the response body. A 201 with `errors: [{ type: "...", message: "No valid users..." }]` means the message was not delivered.

## Provider Message ID

Braze returns a `dispatch_id` in the response:
```json
{
  "dispatch_id": "abc123-def456",
  "errors": [],
  "message": "success"
}
```

The adapter extracts `dispatch_id` as the `providerMessageId` in `SendResult`.
