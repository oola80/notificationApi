# Template Service — Development Changelog

> **Info:** This changelog tracks **code changes only**.
> Design changes are tracked in the root `docs/06-changelog.md`.

## [Unreleased]

### Changed: metaTemplateParameters seed data format from string array to object array (2026-03-02)

- `dbscripts/seed-order-delay-template.sql` — Changed `metaTemplateParameters` from `["customerName", "orderId"]` to `[{"name": "customer_name", "field": "customerName"}, {"name": "order_id", "field": "orderId"}]`. `name` is the Meta placeholder name (snake_case), `field` is the event payload field (camelCase). Supports Meta Cloud API `parameter_name` for named-placeholder templates.

### Added: Technical implementation reference documentation (2026-03-01)

- `tech-template-service-v1.md` — Technical implementation reference documentation

### 2026-03-01 — Fix render() return type to include channelMetadata

- `src/rendering/services/rendering.service.ts` — Added `channelMetadata: Record<string, any>` to `render()` method return type annotation. The method already returned this field in its response object but the type was missing, causing TypeScript compilation failure.

### 2026-03-01 — Include channelMetadata in render response for provider-specific template metadata

- `src/rendering/services/template-cache.service.ts` — Added `channelMetadata: Record<string, any>` to `CompiledTemplate` interface. Cached alongside compiled Handlebars functions.
- `src/rendering/services/rendering.service.ts` — `render()`: reads `channelContent.metadata` from DB and stores as `channelMetadata` in compiled cache entry. Returns `channelMetadata` field in render response alongside `rendered`, `metadata`, and `warnings`. Also updated `preview()` and `warmUp()` to include `channelMetadata` in compiled entries.
- `dbscripts/seed-order-delay-template.sql` — WhatsApp channel insert now includes `metadata` column with `{"metaTemplateName": "order_delay", "metaTemplateLanguage": "es_MX", "metaTemplateParameters": ["customerName", "orderId"]}` JSONB value.
- `src/rendering/services/rendering.service.spec.ts` — Updated all existing `CompiledTemplate` mock objects to include `channelMetadata: {}`. Added 3 new tests: channelMetadata returned with provider metadata, empty object when channel has no metadata, defaults to empty object when metadata is null. Updated warmUp test assertion. Total: 250 tests.

### 2026-02-25 — Phase 5: Metrics, Channel Filter, Sanitization, Render Timeout

- **5 new Prometheus metrics** (`src/metrics/metrics.service.ts`): `ts_template_crud_total` (Counter, `operation` label: create/update/delete/rollback), `ts_template_version_created_total` (Counter), `ts_template_audit_publish_failures_total` (Counter), `ts_template_cache_evictions_total` (Counter), `ts_template_db_pool_active` (Gauge, `collect` callback reads pool stats from DataSource driver). Helper methods: `incrementCrudTotal()`, `incrementVersionCreated()`, `incrementAuditPublishFailure()`, `incrementCacheEviction()`. DataSource injected via `@Optional()` for test compatibility. Total custom metrics: 10.
- **Metrics wiring**: `TemplatesService` — `@Optional() metricsService` injected, calls `incrementCrudTotal` on create/update/delete/rollback, `incrementVersionCreated` on create/update. `AuditPublisherService` — `@Optional() metricsService` injected, calls `incrementAuditPublishFailure` in catch block. `TemplateCacheService` — calls `incrementCacheEviction` on LRU eviction.
- **Channel query filter** (`GET /templates?channel=`): `ListTemplatesQueryDto` — new `channel` field with `@IsIn(['email', 'sms', 'whatsapp', 'push'])` validation. `TemplatesRepository.findAllPaginated()` — `EXISTS (SELECT 1 FROM template_service.template_channels tc WHERE tc.template_version_id = t.current_version_id AND tc.channel = :channel)` subquery applied when `channel` is set.
- **HTML sanitization** (`src/rendering/sanitizers/html-sanitizer.ts`): New file with two pure functions. `sanitizeEmailHtml()` — strips `<script>`, `<iframe>`, `<object>`, `<embed>` tags with content; removes `on*` event handler attributes; replaces `javascript:` URIs; removes CSS `expression()` and non-HTTPS `url()` patterns. `sanitizeTextContent()` — strips all HTML tags for plain text (email subjects). Applied in `RenderingService.render()` and `preview()` for `email` channel only — body sanitized with `sanitizeEmailHtml`, subject sanitized with `sanitizeTextContent`.
- **Render timeout** (`RENDER_TIMEOUT_MS`): `app.config.ts` — `renderTimeoutMs` config (default 5000ms). `env.validation.ts` — `@IsNumber() @IsOptional() RENDER_TIMEOUT_MS`. `errors.ts` — `TS-012` (504 RENDER_TIMEOUT). `RenderingService.render()` — 5 elapsed-time checkpoints (after template lookup, version resolution, channel fetch, variable validation, Handlebars execution) using `process.hrtime.bigint()`. Best-effort pattern: cannot interrupt synchronous Handlebars rendering, but catches slow DB queries and post-render budget violations.
- **Unit tests**: 39 new tests across 2 new suites + updates to 6 existing suites — `html-sanitizer.spec.ts` (21 tests: all sanitization rules, safe HTML passthrough, edge cases), updated `metrics.service.spec.ts` (+5 tests: new metrics registration, helper methods, db pool gauge without dataSource), updated `audit-publisher.service.spec.ts` (+2 tests: audit publish failure metric, no-metrics fallback), updated `template-cache.service.spec.ts` (+1 test: eviction metric), updated `templates.service.spec.ts` (+5 tests: CRUD/version metrics for create/update/delete/rollback, no-metrics fallback), updated `templates.repository.spec.ts` (+2 tests: channel EXISTS subquery applied/not applied), updated `rendering.service.spec.ts` (+3 tests: email sanitization, non-email passthrough, TS-012 timeout). Total: 247 tests across 21 suites.

### 2026-02-25 — Phase 4: RabbitMQ Audit Publishing, E2E Tests & Polish

- **AppRabbitMQModule** (`src/rabbitmq/rabbitmq.module.ts`): Publish-only RabbitMQ module using `@golevelup/nestjs-rabbitmq` `forRootAsync`. Declares `xch.notifications.status` exchange (topic, durable, `createExchangeIfNotExists: false`). No consumers — template service is publish-only. Connection config from ConfigService (rabbitmq.* namespace).
- **RabbitMQ Constants** (`src/rabbitmq/rabbitmq.constants.ts`): `EXCHANGE_NOTIFICATIONS_STATUS` constant. Routing key builders: `templateRoutingKey(action)` → `template.template.{action}`, `renderRoutingKey(status)` → `template.render.{status}`.
- **AuditPublisherService** (`src/rabbitmq/audit-publisher.service.ts`): Fire-and-forget publisher with 6 methods: `publishTemplateCreated`, `publishTemplateUpdated`, `publishTemplateDeleted`, `publishTemplateRolledback`, `publishRenderCompleted`, `publishRenderFailed`. All publishes wrapped in try-catch (log warning, never throw). Message payload: `{ eventType, timestamp, service: 'template-service', correlationId?, data }`. Persistent messages with custom headers (`x-service`, `x-event-type`, `x-correlation-id`).
- **Audit publishing integration**: `TemplatesService.create()` → `publishTemplateCreated`, `update()` → `publishTemplateUpdated`, `delete()` → `publishTemplateDeleted`, `rollback()` → `publishTemplateRolledback`. `RenderingService.render()` → `publishRenderCompleted` on success, `publishRenderFailed` on error. All injected via `@Optional()` for backward compatibility.
- **Rollback endpoint** (`POST /templates/:id/rollback`): New `RollbackTemplateDto` (versionNumber required, updatedBy optional). `TemplatesService.rollback()` validates target version exists and is different from current, updates `currentVersionId` pointer, invalidates cache. New error code `TS-011` (400 INVALID_ROLLBACK_VERSION) for invalid or same-version rollback attempts.
- **Enhanced health check** (`src/health/health.controller.ts`): Added RabbitMQ health indicator (Management API `/api/healthchecks/node` with Basic auth, 5s timeout). Added cache status (`size`, `maxSize`) via `TemplateCacheService.getStats()`. Three-state status: `healthy` (DB+RMQ up), `degraded` (DB up, RMQ down), `unhealthy` (DB down). `HealthModule` now imports `RenderingModule` for `TemplateCacheService` access.
- **Module wiring**: `AppRabbitMQModule` imported in `AppModule` (between MetricsModule and HealthModule), `TemplatesModule` (for TemplatesService audit publishing), and `RenderingModule` (for RenderingService audit publishing).
- **E2E test infrastructure**: `test/test-utils.ts` with `createTestApp()` (mirrors main.ts global middleware) and `cleanupTestData()` helper. `test/jest-e2e.json` updated with `moduleNameMapper` for ESM compat and 30s timeout.
- **E2E tests**: `app.e2e-spec.ts` (health + metrics), `templates.e2e-spec.ts` (full CRUD lifecycle: create with multiple channels, duplicate slug 409, invalid Handlebars 422, list with pagination, search filter, get by ID, get non-existent 404, update with versioning, soft delete), `rendering.e2e-spec.ts` (render with valid data, inactive template 422, version-specific render, SMS length warnings, preview all channels), `rollback.e2e-spec.ts` (v2→v1 rollback, non-existent version 400, same version 400, non-existent template 404).
- **Unit tests**: 29 new tests across 2 new suites + updates to 3 existing suites — `audit-publisher.service.spec.ts` (13 tests: all publish methods, fire-and-forget error handling, correlationId propagation), `rabbitmq.constants.spec.ts` (4 tests). Updated `health.controller.spec.ts` (9 tests: healthy/degraded/unhealthy states, RabbitMQ check, cache status). Updated `templates.controller.spec.ts` (+1 rollback test). ### 2026-02-25 — Phase 3: Rendering Engine, Caching & Preview

- **RenderingModule** (`src/rendering/`): Self-contained module with `RenderingController`, `RenderingService`, `TemplateCacheService`, and custom Handlebars helpers. Exports `TemplateCacheService` for cache invalidation by `TemplatesModule`.
- **Render endpoint** (`POST /templates/:id/render`): Full rendering pipeline — template lookup, version resolution (explicit or current), LRU cache lookup/compile, required variable validation (TS-006), Handlebars execution with error handling (TS-005), output length warnings per channel (SMS >160/>1600, WhatsApp >4096, push >256), Prometheus metrics (render duration histogram, render total counter). Returns `{ rendered: { subject?, body }, metadata, warnings }`.
- **Preview endpoint** (`POST /templates/:id/preview`): Renders all channels for a template version in one call. Returns `{ previews: [{ channel, subject?, body, warnings }], metadata }`.
- **TemplateCacheService** (`src/rendering/services/template-cache.service.ts`): LRU in-memory cache for compiled Handlebars `TemplateDelegate` functions. Cache key: `{templateId}:{versionId}:{channel}`. Configurable max size via `TEMPLATE_CACHE_MAX_SIZE` env var (default 1000). Metrics-instrumented: cache hit/miss counters, cache size gauge. Methods: `get`, `set`, `invalidate(templateId)`, `invalidateAll`, `getStats`.
- **Custom Handlebars helpers** (`src/rendering/helpers/handlebars-helpers.ts`): 8 helpers registered once at module init — `formatCurrency` (Intl.NumberFormat), `formatDate` (dayjs), `uppercase`, `lowercase`, `truncate` (with ellipsis), `eq` (block), `gt` (block), `default` (fallback).
- **Cache warm-up**: `RenderingService` implements `OnApplicationBootstrap` — pre-compiles all active templates' current version channels into the cache on startup.
- **Cache invalidation**: `TemplatesService.update()` and `delete()` now call `TemplateCacheService.invalidate(id)` to evict stale cached compilations after template changes. `TemplateCacheService` injected via `@Optional()` for backward compatibility with existing tests.
- **Configuration** (`src/config/app.config.ts`): Added `cacheMaxSize`, `smsWarnLength`, `smsMaxLength`, `whatsappMaxLength`, `pushMaxLength` config values with env var overrides.
- **DTOs**: `RenderTemplateDto` (channel, data, versionNumber?), `PreviewTemplateDto` (data, versionNumber?) with class-validator decorators.
- **Module registration**: `RenderingModule` imported in `AppModule` (after `TemplatesModule`) and in `TemplatesModule` (for `TemplateCacheService` access).
- **Unit tests**: 62 new tests across 4 suites — `handlebars-helpers.spec.ts` (17 tests), `template-cache.service.spec.ts` (12 tests), `rendering.service.spec.ts` (20 tests), `rendering.controller.spec.ts` (7 tests). Total: 179 tests across 18 suites.
- **Endpoints**: Updated `endpoints-template-service.md` — render and preview endpoints marked as Done.

### 2026-02-25 — Phase 2: Template CRUD & Versioning

- **Entities** (`src/templates/entities/`): 4 TypeORM entities mapping to DB tables — `Template` (OneToMany versions/variables), `TemplateVersion` (ManyToOne template, OneToMany channels), `TemplateChannel` (ManyToOne version), `TemplateVariable` (ManyToOne template). All use snake_case DB column names with camelCase TypeScript properties.
- **DTOs** (`src/templates/dto/`): `CreateTemplateDto` (slug, name, description, channels[], createdBy), `UpdateTemplateDto` (channels[], changeSummary, updatedBy), `ListTemplatesQueryDto` (page, limit, search, isActive, sortBy, sortOrder with boolean transform), `ChannelDto` (channel enum [email/sms/whatsapp/push], conditional subject validation for email/push, body, optional metadata). Barrel export via `index.ts`.
- **Repositories** (`src/templates/repositories/`): `TemplatesRepository` extends `PgBaseRepository` with paginated listing (ILIKE search on slug/name, isActive filter, configurable sort), `findByIdWithRelations` (eager loads versions→channels + variables), `existsBySlug` (with excludeId support). `TemplateVersionsRepository` (next version number via MAX query, find by template). `TemplateChannelsRepository` (batch create, find by version). `TemplateVariablesRepository` (upsert: diff existing vs detected, remove stale, add new).
- **VariableDetectorService** (`src/templates/services/variable-detector.service.ts`): Parses Handlebars AST via `Handlebars.parse()`, walks `MustacheStatement` (simple vars + helper params), `BlockStatement` (#if/#each/#unless params + body/inverse), and `PathExpression` nodes. Extracts variable names including nested paths (e.g., `order.id`). Deduplicates across all channels. Excludes built-in helpers (formatCurrency, formatDate, uppercase, lowercase, truncate, eq, gt, default, if, each, unless, with, lookup, log) and @data variables.
- **TemplatesService** (`src/templates/services/templates.service.ts`): Full CRUD pipeline — `create()`: validate slug uniqueness → validate Handlebars syntax via `precompile()` → transactional creation (template + version 1 + channels + variables) → return assembled response. `findAll()`: paginated list delegating to repository. `findById()`: full template with all relations (TS-009 on not found). `update()`: validate existence → validate Handlebars syntax → transactional new version (increment version number, create channels, upsert variables, update `current_version_id` pointer) → reload with relations. `delete()`: soft delete (set `isActive = false`). All create/update use `DataSource.transaction()`.
- **TemplatesController** (`src/templates/templates.controller.ts`): Thin controller with 5 endpoints — `POST /templates` (201), `GET /templates` (200), `GET /templates/:id` (200), `PUT /templates/:id` (200), `DELETE /templates/:id` (200). `ParseUUIDPipe` on `:id` params.
- **TemplatesModule** (`src/templates/templates.module.ts`): Registers all 4 entities via `TypeOrmModule.forFeature`, provides all repositories + services, exports `TemplatesService`.
- **AppModule** (`src/app.module.ts`): Added `TemplatesModule` import.
- **Unit tests**: 61 new tests across 4 suites — `templates.controller.spec.ts` (5 endpoint tests), `templates.service.spec.ts` (16 tests: create pipeline, slug conflict TS-002, Handlebars validation TS-003, findAll delegation, findById with relations, findById not found TS-009, update versioning, update version increment, update variable upsert, delete soft delete, delete not found), `variable-detector.service.spec.ts` (19 tests: simple vars, nested paths, #if/#each/#unless, subject+body, deduplication, helper exclusion, @data exclusion, this exclusion, plain text, sorted results, else blocks, all built-in helpers), `templates.repository.spec.ts` (16 tests: search filter, isActive filter, default/custom sort, pagination, relations, slug existence checks, save/create delegation). Total: 93 tests across 11 suites.
