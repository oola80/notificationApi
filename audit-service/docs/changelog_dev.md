### Phase 4 — DLQ Monitoring, Analytics Engine & Data Retention (2026-02-28)

- **DLQ CRUD endpoints** — `GET /audit/dlq` with paginated query, 4 optional filters (status, originalQueue, from, to), response includes `meta.statusCounts` (aggregated count per status). `PATCH /audit/dlq/:id` for status updates with transition validation (pending → investigated, investigated → reprocessed|discarded, pending → discarded); sets `resolved_at`/`resolved_by` on terminal states; rejects invalid transitions with AUD-006. `POST /audit/dlq/:id/reprocess` republishes DLQ entry payload to original exchange/routing key; validates entry is in `investigated` status; marks as `reprocessed` with resolved metadata.
- **DlqEntriesRepository enhanced** — 3 new methods: `findWithFilters()` (QueryBuilder with 4 optional WHERE clauses + date range), `statusCounts()` (aggregate COUNT grouped by status), `updateEntry()` (partial update by ID).
- **DlqPublisher** — New RabbitMQ publisher service for DLQ reprocessing (single outbound publish use case). Wraps `AmqpConnection.publish()`. Added to `AppRabbitMQModule` providers and exports.
- **Analytics query endpoints** — `GET /audit/analytics` with required from/to params, optional period (hourly/daily, default daily), channel, eventType filters. Returns `{ data, meta: { period, from, to, totalRecords } }`. `GET /audit/analytics/summary` returns dashboard summary: `today` (totalSent, totalDelivered, deliveryRate, totalFailed, failureRate), `last7Days` (totalSent, totalDelivered, deliveryRate, avgLatencyMs), `channelBreakdown` (per-channel sent/delivered/rate).
- **Aggregation service** — NestJS `@Cron` scheduled jobs via `@nestjs/schedule`. Hourly rollup (default `5 * * * *`) computes metrics for previous full hour. Daily rollup (default `15 0 * * *`) computes metrics for previous full day. Groups delivery_receipts by channel, counts by status (sent/delivered/failed/opened/clicked/bounced), merges suppression counts from audit_events, produces per-channel rows + `_all` cross-channel totals with `event_type = NULL` (cross-type). Idempotent upsert via `ON CONFLICT (period, period_start, channel, COALESCE(event_type, ''))`.
- **NotificationAnalyticsRepository enhanced** — 5 new methods: `findWithFilters()` (QueryBuilder with period/date/channel/eventType), `upsertRow()` (raw SQL idempotent upsert), `aggregateFromReceipts()` (aggregate delivery_receipts by channel using FILTER), `countSuppressed()` (count suppression events from audit_events per channel), `findForSummary()` (query for summary computation).
- **Data retention service** — `RetentionModule` with `RetentionService`. Payload purge `@Cron('0 3 * * *')` calls `purge_audit_payloads()` PL/pgSQL function with configurable `RETENTION_PAYLOAD_DAYS` (default 90). DLQ cleanup `@Cron('30 3 * * *')` hard-deletes `dlq_entries` in terminal states (reprocessed/discarded) older than 90 days; does NOT delete pending or investigated entries.
- **Database script updated** — Replaced `idx_analytics_period_start` unique index with `idx_analytics_upsert_key` using `COALESCE(event_type, '')` for NULL-safe upsert. Added `idx_dlq_entries_captured_at` index for DLQ date range queries.
- **Dependencies** — Added `@nestjs/schedule` for cron job support. `ScheduleModule.forRoot()` imported in `AnalyticsModule`.
- **Modules updated** — `DlqModule` now includes `DlqController`, `DlqService`, imports `AppRabbitMQModule`. `AnalyticsModule` now includes `AnalyticsController`, `AnalyticsService`, `AggregationService`, `ScheduleModule`. `AppRabbitMQModule` now provides and exports `DlqPublisher`. `AppModule` imports `RetentionModule`. `NotificationAnalytics` entity index updated (removed incorrect unique composite, actual uniqueness enforced by DB functional index).
- **Unit tests** — 74 new tests across 7 new suites: dlq-entries.repository.spec.ts (+10), dlq.service.spec.ts (18), dlq.controller.spec.ts (4), dlq-publisher.service.spec.ts (3), aggregation.service.spec.ts (8), analytics.service.spec.ts (9), analytics.controller.spec.ts (2), notification-analytics.repository.spec.ts (+10). **Total: 338 unit tests across 41 suites.**
- **E2E tests** — 21 new tests across 2 new files: dlq.e2e-spec.ts (13 — list with filters, status update lifecycle, reprocess with mock RabbitMQ verification, invalid transitions rejected), analytics.e2e-spec.ts (8 — query with filters, default period, missing params validation, summary with today/7d/channels, empty data). Updated test-utils.ts with `createMockDlqPublisher()`, enhanced `createMockDlqEntriesRepository()` and `createMockNotificationAnalyticsRepository()`. **Total: 58 E2E tests across 8 suites.**

### Phase 3 — REST Query API: Logs, Search, Trace, Receipts (2026-02-28)

- **Audit Logs endpoint** — `GET /audit/logs` with paginated query, 7 optional filters (notificationId, correlationId, cycleId, eventType, actor, from, to), inline full-text search via `q` param, 90-day date range validation (AUD-004). Response shape: `{ data, meta: { page, pageSize, totalCount, totalPages } }`.
- **Full-text search endpoint** — `GET /audit/search` with required `q` param, automatic operator detection (`&`, `|`, `!`, `(` → `to_tsquery`, else `plainto_tsquery`), `ts_rank_cd` ranking, date range filtering, `searchMaxResults` config enforcement (AUD-007), search duration metric observation.
- **Notification trace endpoint** — `GET /audit/trace/:notificationId` with 5-step algorithm: fetch events, fetch receipts, merge by timestamp, extract headers from earliest event, build summary. Returns `{ summary, timeline }` with merged `audit_event` + `delivery_receipt` entries sorted chronologically. 404 → AUD-008.
- **Correlation trace endpoint** — `GET /audit/trace/correlation/:correlationId` finds distinct notification IDs, builds individual traces, returns grouped `{ correlationId, notifications }`. 404 → AUD-008.
- **Cycle trace endpoint** — `GET /audit/trace/cycle/:cycleId` same grouping pattern as correlation trace. 404 → AUD-008.
- **Receipts endpoint** — `GET /audit/receipts/:notificationId` returns `{ notificationId, receipts }`. 404 → AUD-008.
- **AuditEventsRepository enhanced** — 4 new methods: `findWithFilters()` (QueryBuilder with 7 optional WHERE clauses + date range + inline FTS), `fullTextSearch()` (QueryBuilder with `ts_rank_cd` ranking, parameterized tsquery, date filtering), `findByNotificationIdOrdered()`, `findDistinctNotificationIds()`.
- **DeliveryReceiptsRepository enhanced** — 2 new methods: `findByNotificationIdOrdered()`, `findByNotificationId()`.
- **New modules** — SearchModule (imports EventsModule), TraceModule (imports EventsModule + ReceiptsModule). EventsModule updated with AuditLogsController + AuditLogsService. ReceiptsModule updated with AuditReceiptsController + AuditReceiptsService.
- **DTOs** — `ListAuditLogsQueryDto` (10 validated fields with defaults), `SearchQueryDto` (required q, optional date range + pagination).
- **Interfaces** — `TimelineEntry`, `NotificationTraceSummary`, `NotificationTraceResponse`, `CorrelationTraceResponse`, `CycleTraceResponse`.
- **Unit tests** — 86 new tests across 8 new suites: audit-events.repository.spec.ts (+15), audit-logs.service.spec.ts (12), audit-logs.controller.spec.ts (2), search.service.spec.ts (12), search.controller.spec.ts (2), trace.service.spec.ts (18), trace.controller.spec.ts (4), delivery-receipts.repository.spec.ts (+4), audit-receipts.service.spec.ts (4), audit-receipts.controller.spec.ts (2). **Total: 264 unit tests across 34 suites.**

### Phase 2 — RabbitMQ Consumer Pipeline with Batch Insert (2026-02-27)

- **RabbitMQ module** — `AppRabbitMQModule` with `@golevelup/nestjs-rabbitmq` v7.1.2, consumer-only setup (no publisher), 4 exchanges (`xch.events.normalized`, `xch.notifications.deliver`, `xch.notifications.status`, `xch.notifications.dlq`), 5 consumed queues, 5 named channels with per-queue prefetch (events:20, deliver:20, status:15, template:10, dlq:5), heartbeat 30s, reconnect 5s.
- **RabbitMQ constants** — Exchange, queue, channel, routing key constants, `ALL_CONSUMED_QUEUES` array, `extractProviderFromWebhookKey()` and `isWebhookRoutingKey()` helper functions.
- **BatchBufferService** — Generic batch buffer accumulating messages up to `CONSUMER_BATCH_SIZE`, flushing on size or timer (`CONSUMER_FLUSH_INTERVAL_MS`). Deferred ACK/NACK via Promises (handler returns `Promise<void | Nack>` that resolves when batch flushes). Transient error detection by PostgreSQL error codes (08000, 08003, 08006, 40001, 40P01, 55P03, 57014) and message patterns (Connection, timeout, ECONNREFUSED, ECONNRESET). NACK+requeue on transient failure, ACK+discard on permanent error, poison message handling after `CONSUMER_MAX_RETRIES` with in-memory fingerprint-based retry tracking. Graceful shutdown via `OnModuleDestroy`.
- **EventsConsumer** — Consumes `audit.events` (routing key `event.#`). Maps messages to `EVENT_INGESTED`/`EVENT_NORMALIZED`/`EVENT_VALIDATION_FAILED`/`EVENT_DUPLICATE_DETECTED` based on auditEventType, status, validationErrors, isDuplicate, and normalizedPayload fields. Actor: `event-ingestion-service`. Batch inserts into `audit_events`.
- **DeliverConsumer** — Consumes `audit.deliver` (routing key `notification.deliver.#`). Maps all messages to `DELIVERY_DISPATCHED`. Actor: `notification-engine-service`. Batch inserts into `audit_events`.
- **StatusConsumer** — Consumes `q.status.updates` with dual routing keys (`notification.status.#`, `adapter.webhook.#`). Delivery-path messages map to `DELIVERY_ATTEMPTED`/`SENT`/`FAILED`/`RETRYING`; creates `DeliveryReceipt` only when status=sent and providerMessageId present. Webhook messages map to `DELIVERED`/`BOUNCED`/`OPENED`/`CLICKED`/`UNSUBSCRIBED`/`SPAM_COMPLAINT`; always creates `DeliveryReceipt`; detects orphaned receipts (no notificationId). Uses compound `StatusBufferRecord` for batch inserts into `audit_events` + `delivery_receipts`.
- **TemplateConsumer** — Consumes `q.audit.template` (routing key `template.#`). Maps routing key action to `TEMPLATE_CREATED`/`UPDATED`/`DELETED`/`ROLLEDBACK`. Actor: `template-service`. Sets notificationId to templateId.
- **DlqConsumer** — Consumes `audit.dlq` (fanout). Extracts `x-death` headers (queue, exchange, routing-keys, reason, count). Creates both `DlqEntry` (status: pending) and `AuditEvent` (type: `DLQ_CAPTURED`). Uses compound `DlqBufferRecord` for batch inserts into `audit_events` + `dlq_entries`.
- **ConsumersModule** — Wires all 5 consumers + BatchBufferService, imports AppRabbitMQModule, EventsModule, ReceiptsModule, DlqModule.
- **PgBaseRepository.insertMany()** — Added bulk insert method using TypeORM `insert()` for efficient batch INSERTs (vs N individual `save()` calls).
- **ConsumerHealthIndicator** — Checks `AmqpConnection.managedChannel` for connection status, fetches per-queue message depths from RabbitMQ Management API, updates `audit_consumer_lag` gauge.
- **Health controller updated** — Readiness endpoint now includes 4 checks: database, rabbitmq, dlqDepth, consumers (was 3).
- **Metrics updated** — Added `audit_poison_messages_total` counter (labels: queue). Now 15 custom metrics (6 counters, 5 histograms, 4 gauges).
- **Unit tests** — 100 new tests across 8 new suites: rabbitmq.constants.spec.ts (11), batch-buffer.service.spec.ts (13), events.consumer.spec.ts (10), deliver.consumer.spec.ts (9), status.consumer.spec.ts (16), template.consumer.spec.ts (10), dlq.consumer.spec.ts (11), consumer-health.indicator.spec.ts (4). Updated health.controller.spec.ts (+2), metrics.service.spec.ts (+2). **Total: 178 unit tests across 26 suites.**
- **E2E tests** — 8 new tests in consumers.e2e-spec.ts (batch flush, timer flush, deserialization errors, status consumer with dual routing keys, DLQ consumer with x-death extraction). Updated app.e2e-spec.ts with ConsumerHealthIndicator mock. **Total: 13 E2E tests across 2 suites.**
